# CLAUDE.md

减少常见 LLM 编码错误的行为指南。可根据需要与项目专属指令合并使用。

**权衡取舍：** 本指南偏向谨慎而非速度。对于琐碎任务，自行判断即可。

## 1. 先思考再编码

**不要假设。不要隐藏困惑。暴露权衡。**

实施之前：
- 明确陈述你的假设。如有不确定，直接问。
- 如果存在多种解读，全部列出——不要默默替用户选。
- 如果有更简单的方案，直接说出来。必要时可以据理反驳。
- 如果某件事不清楚，停下来。指出哪些地方令人困惑。然后问。

## 2. 简洁优先

**解决问题所需的最少代码。不写任何推测性的东西。**

- 不实现需求之外的特性。
- 不为一次性代码抽象化。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果你写了 200 行而实际只需要 50 行，重写它。

问问自己："资深工程师会觉得这个过于复杂吗？"如果是，请简化。

## 3. 外科手术式修改

**只动必须动的。只清理你自己留下的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的代码。
- 匹配现有风格，即使你有不同做法。
- 如果发现无关的死代码，提一句——不要删掉。

当你的修改产生孤儿代码时：
- 删除因**你的修改**而变得未使用的 import、变量或函数。
- 除非被要求，否则不要删除已有的死代码。

检验标准：每一处修改的行都应能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 Bug" → "编写能复现它的测试，然后让它们通过"
- "重构 X" → "确保重构前后测试全部通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

强成功标准能让你独立循环推进。弱成功标准（"让它能跑就行"）需要不断确认澄清。

---

**这些指南生效的标志是：** diff 中出现更少不必要的修改，因过度复杂而重写的次数减少，以及确认性问题出现在实施之前而非犯错之后。
